# ROADMAP.md — MVP 路线图

> 本路线图围绕一个**降级后的核心假设**（详见 decisions/0001 F3）：
> **AI 能从日常小事中识别孩子的"重复模式 + 成长变化轨迹"，并据此给出对父母有价值的视角**。
>
> "天赋识别"留给 v2——学龄前的信号噪音过高、心理学上对终生天赋几乎无预测效度。MVP 阶段先验证可量化的"模式 + 变化"。
> 8 周内必须验证这个降级版假设是否成立。如果验证失败，整个项目要回到 VISION 重新审视方向。

### 关于"成长变化"的特别说明
孩子的发展是连续的而非静态的。同一个行为在不同阶段含义完全不同：
- 瑶瑶 2.5 岁喜欢"一个一个数台阶"和 4 岁还在数完全是两件事——前者是数感萌芽，后者可能值得追问
- "兴趣消失"不一定是失去兴趣，可能只是发展阶段过了
- 因此信号系统必须同时记录 (1) 模式强度 (2) 出现时孩子的发展月龄/阶段 (3) 与同月龄的对照基线

任何一处把"重复信号"理解成"恒定特质"就走偏了。

## 0. 北极星 (North Star)

**MVP 阶段先把产品跑起来，北极星指标延后到 v0.2 再正式定义**（详见 decisions/0001 F2）。

8 周内的暂定验证标准（用于自检，不作为对外正式 KPI）：
- 你打开 APP 收到的第一份周报里，至少有 1 条洞察让你说出"诶，我没注意到这一点"
- 信号系统能正确识别瑶瑶过去 30 天里至少 3 个"重复模式"或"明显变化"

正式的北极星会等真正的产品体验出来后再讨论——避免在没有产品的情况下空想指标。

## 1. 阶段划分

### Phase 0：架构基线（1 周）
**目标**：跑通"对话→结构化记录→存储→检索"的最短链路

里程碑：
- [x] M0.1 本地服务骨架 (Python + FastAPI)
- [x] M0.2 Ollama 接入，记录 Agent 能从中文口语生成结构化条目
- [x] M0.3 SQLite + sqlite-vec 跑通最小读写
- [x] M0.4 Web 端最简界面：能输入对话、看到结构化结果（**已合并到 Phase 1：Next.js + 三屏 `/log` `/timeline` `/heatmap` 见 `web/`**）

**模式建议**：Cowork 出 PRD → Code 实现
**完成判定**：你能说出"今天瑶瑶第一次自己尿尿了"，看到结构化条目落库。
**实际验证**：`POST /events` + `GET /events` 真机跑通，valid 9/10 / type-match 9/10（≥ F7 门槛 9/8）。详见 `reports/phase0-baseline.md`。

---

### Phase 1：信号体系（2 周）
**目标**：把"原始事件"变成"信号"，搭起**重复模式 + 成长变化轨迹**的最小骨架

里程碑：
- [x] M1.1 定义信号 schema：
       - 类型：兴趣模式 / 情绪模式 / 技能模式 / 异常变化 / 成长跃迁
       - 每个信号必须带 (强度, 出现时孩子月龄, 与上一周期的变化量)
- [x] M1.2 信号提取 Agent v0：基于规则 + 本地模型混合
- [x] M1.3 信号可视化：时间轴 + 热度图，**且热度图必须能展示"随发展阶段的变化"**
       （而不是简单"全年累计强度"）
- [x] M1.4 数据回灌：你提供过去若干月的回忆事件，验证信号系统能识别

**模式建议**：Cowork 设计 schema、Code 实现、你做数据回灌
**完成判定**：你看着信号热度图能说"对，瑶瑶最近确实对音乐有兴趣"，**并且能区分出哪些是稳定模式、哪些是阶段性的"过去就过去了"**。

---

### Phase 2：洞察周报（2 周）
**目标**：每周自动生成一份周报，验证云端洞察 Agent 的输出质量

里程碑：
- [ ] M2.1 上下文压缩 Agent：把一周事件压到 ≤ 4k tokens
- [ ] M2.2 洞察 Agent v0：调 Claude 生成周报草稿
- [ ] M2.3 周报模板（亮点/趋势/下周关注/开放问题）
- [ ] M2.4 反馈按钮（每条洞察 👍/👎/不准确）
- [ ] M2.5 第一份真实周报送到你眼前

**模式建议**：Cowork 设计 prompt + 模板，Code 实现，你做评分
**完成判定**：连续 2 周收到周报，至少 1 条洞察被你标"准确且有价值"。

---

### Phase 2.5：家庭手机内测（1-2 周）
**目标**：把本机实验室里的 MVP 变成邀请制 ≤10 个家庭能在华为/安卓手机上试用的内测产品。

里程碑：
- [x] M2.5.1 数据库演进 ADR + 家庭内测 PRD 收口
- [ ] M2.5.2 服务端 Postgres + migration 机制
- [ ] M2.5.3 最小家庭访问控制（访问码 / magic link 二选一）
- [ ] M2.5.4 手机 Web / PWA 体验优化
- [ ] M2.5.5 公网 HTTPS 部署 + 家庭真机 smoke test

**模式建议**：Cowork/作者确认家庭内测边界，Code 实现部署与数据层。
**完成判定**：家庭成员不需要打开终端，只用手机浏览器即可记录事件、查看时间轴/周报，并且真实数据不进入 Git、不困在作者某台电脑的 SQLite 文件里。

---

### Phase 3：培养建议 + 知识库（2 周）
**目标**：用知识库 RAG 把洞察转化为可执行建议

里程碑：
- [ ] M3.1 育儿知识库 v0（精选 30-50 段经典/前沿育儿理论的 RAG 块）
- [ ] M3.2 培养建议 Agent v0：基于信号 + RAG 给 3 条不同方向的建议
- [ ] M3.3 **多元反馈追踪**（已按 decisions/0001 F1 调整）：
       - "采纳"标记保留作辅助参考
       - 同时记录："父母收到建议后是否产生新讨论"、"父母是否做出与建议不同的新决定"
       - 也记录被动信号：是否打开周报、阅读时长、是否复述给家人
       - 8 周后一起看哪个指标更能反映"建议有价值"，再决定 v0.2 的主指标
- [ ] M3.4 文化适配：所有建议必须用中文家庭语境表达

**模式建议**：Cowork 主导知识库挑选 + 写 prompt，Code 做检索引擎
**完成判定**：你收到的建议里至少 2 条触发了你的"主动思考或讨论"（不限于"采纳"），并在 1 周后能给出反馈。

---

### Phase 4：反馈闭环 & 自动小迭代（1 周）
**目标**：让你的反馈能自动触发产品改进

里程碑：
- [ ] M4.1 反馈分类器：把反馈分为「文案」「阈值」「逻辑」「需求」四类
- [ ] M4.2 自动小改：文案 + 阈值类直接由产品 Agent 生成 PR-style diff
- [ ] M4.3 大改进 backlog：需求 + 逻辑改动写成 PRD 进 `prd/`
- [ ] M4.4 试运行：你提 5 条反馈，看自动化处理覆盖率

**模式建议**：Cowork 设计分类器 prompt，Code 实现 diff 生成
**完成判定**：5 条反馈中至少 3 条被自动正确分类，1 条直接生成可合并的代码改动。

## 3. 后 MVP（v1+）

按价值优先级排（不在 8 周里做）：

1. **iOS 原生 APP**（移动端体验，不靠 Web）
2. **语音直接录入**（端上 Whisper.cpp）
3. **配偶/家庭成员协同账户**
4. **图片理解**（孩子画作的视觉信号）
5. **教练匹配 Agent**（供给端要先建立）
6. **跨代视图**（专门的"15年后给孩子看"模式）
7. **多孩子支持**

## 4. 验证失败的应对预案

如果 8 周后北极星问题"AI 给出过让你意外的洞察了吗"答案是 **No**：

- 第一反思：是信号体系不够细，还是云端 prompt 不够好，还是知识库没用上？
- 第二反思：是不是单用户数据量不够，需要扩大早期用户？
- 第三反思：是不是"AI 识别天赋"本身在当前技术下还做不到？需不需要把 MVP 假设换成"长期记忆陪伴"或"父母自我反思"？

把这三层问题留在每个 Phase 末尾的 retro 里讨论。
