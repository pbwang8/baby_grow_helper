# ARCHITECTURE.md — 技术架构 & 多Agent协同 & 成本模型

> 本文档定义"产品本身怎么跑"。Code 模式实现时以这份为准。
> 任何架构变更，必须先在 `decisions/` 写一份 ADR 并经用户确认。

---

## 1. 设计原则（按优先级）

1. **本地优先 / 云端兜底**：日常记录、检索、轻量提炼跑在用户设备；只在重大判断时调用云端大模型
2. **数据归用户所有**：默认数据在本地（设备），云只在必要时短暂传输
3. **可中断、可重跑**：任何 Agent 都能被中断，重跑不会污染记忆
4. **每个 Agent 都是纯函数式输入→输出**：方便测试、回放、替换
5. **预算硬约束**：单用户运行成本必须可控，月度调用上限可配置

## 3. 本地 Agent 详细职责

### 3.1 记录 Agent（Recorder）
**输入**：用户语音/文字（"今天瑶瑶第一次自己尿尿了，特别兴奋还跳起来庆祝"）
**输出**：结构化条目
```json
{
  "id": "evt_2026_05_15_001",
  "timestamp": "2026-05-15T19:42:00+08:00",
  "child": "瑶瑶",
  "raw_text": "今天瑶瑶第一次自己尿尿了，特别兴奋还跳起来庆祝",
  "summary": "首次自主如厕成功",
  "type": "milestone",
  "domain": ["self_care", "independence"],
  "emotion": ["proud", "excited"],
  "context": "evening, parent observing"
}
```
**模型**：本地 4-8B 量级小模型（Phi-3-mini / Qwen2.5-3B），完全在设备上跑，不联网

### 3.2 信号提取 Agent（Signal Extractor）
**触发**：每天晚上一次（或每收集 N 条新记录后）
**职责**：扫描最近窗口的记录，识别值得标记的"信号"
- 兴趣信号：在某个领域的持续关注（连续 N 天、强度 M 以上）
- 情绪信号：在特定情境下的反复情绪反应
- 技能信号：能力突破点（first-time / level-up）
- 异常信号：和历史模式不一致的行为

**输出**：写入 `signals` 表，标注 `confidence` 和 `evidence_event_ids`

### 3.3 检索 Agent（Retriever）
**职责**：当其他 Agent 或用户需要历史上下文时，提供精准检索
- 时间检索：上周/上月/同期
- 主题检索："瑶瑶对音乐的反应"
- 关联检索：和当前事件相似的历史事件

向量库：`sqlite-vec` 或 `chromadb`（嵌入模型用本地 BGE-small）

## 4. 云端 Agent 详细职责

### 4.1 洞察 Agent（Insight）
**触发**：每周日晚自动 / 用户手动请求 / 重大里程碑
**输入**：本地组装好的"上下文包"——本周事件摘要 + 信号变化 + 历史对比基线
**输出**：一份周报，结构包括：
- 本周亮点（3 条）
- 趋势观察（兴趣方向、情绪规律、能力发展）
- 一个值得在下周关注的视角
- 一个开放性问题给父母（不直接给答案）

**关键约束**：周报字数压缩，单次调用 token 受控（详见 §6）

### 4.2 培养建议 Agent（Coach）
**触发**：用户主动询问"该怎么做"，或洞察 Agent 觉得有"高置信度建议"
**输入**：当前问题 + 该孩子最近 30-90 天画像 + 育儿知识库 RAG 结果
**输出**：3 个不同方向的建议（不是单一答案），并标注每条建议的依据

**知识库 RAG**（独立子系统）：
- 经典：蒙台梭利、皮亚杰、维果茨基、华德福、瑞吉欧、儿童依恋理论
- 前沿：执行功能 (EF)、心智化、成长型思维、感觉统合、神经多样性
- 文化适配：中文家庭情境的本土化解读
- 来源标注：每条建议必须能溯源

## 5. 数据模型（SQLite, 本地）

```sql
-- 孩子档案
children(id, name, birthday, profile_blob_json)

-- 原始事件（对话记录归档后的结构化条目）
events(id, child_id, timestamp, raw_text, summary,
       type, domains_json, emotions_json, context, source)

-- 信号
signals(id, child_id, signal_type, domain, confidence,
        first_seen_at, last_seen_at, evidence_event_ids_json,
        status -- active/dormant/dismissed
)

-- 洞察（周/月报）
insights(id, child_id, period_start, period_end, kind,
         body_md, source_signal_ids_json, model_used, generated_at)

-- 用户对洞察/建议的反馈（用于反馈循环）
feedback(id, target_kind, target_id, rating, note, created_at)

-- 决策与建议追溯
recommendations(id, insight_id, body_md, evidence_json,
                user_action_json -- accepted/dismissed/snoozed
)

-- 嵌入向量（与 events 通过 id 关联）
event_embeddings(event_id PRIMARY KEY, vector BLOB)
```

## 6. 成本模型（$200/月预算）

### 6.1 假设
- MVP 阶段单用户（你自己）
- 每天对话条目 ≈ 5 条，每条 ≈ 80 tokens
- 周报每周一次
- 偶发深度建议每周 1-2 次

### 6.2 估算
| 调用 | 频率 | 单次 token | 月 token | 月成本 (Claude Sonnet 大致单价) |
|---|---|---|---|---|
| 本地记录 + 信号提取 | 每天 | 0 (本地) | 0 | **$0** |
| 周报洞察 (输入压缩到 4k, 输出 1.5k) | 4 次/月 | ~5500 | 22k | < $1 |
| 深度建议 (RAG 之后输入 6k, 输出 2k) | 8 次/月 | ~8000 | 64k | < $2 |
| 知识库索引/嵌入（一次性） | 1 次 | — | — | < $5（一次性） |

**单用户月成本估算：< $5**

→ $200/月可覆盖 **40-50 个早期用户**，足够 MVP + 小范围内测。

### 6.3 控成本红线
1. 永远不让"对话Agent"直接挂在云端大模型上当客服——那是成本黑洞
2. 任何云端调用必须先经过本地"上下文压缩 Agent"——把过去 7 天事件压成 ≤ 4k tokens 再发
3. 每个用户每月有 hard cap（默认 100k tokens），到顶就降级提示
4. 嵌入用本地模型，不调云端 embedding API

## 7. 技术选型建议（待用户/Code 评审）

### MVP（前 8 周）
- **后端语言**：Python（生态最适合 Agent 编排 + Ollama 接入）
- **本地模型**：Ollama + Qwen2.5-3B-Instruct（中文友好、设备占用低）
- **嵌入**：BGE-small-zh（本地）
- **存储**：SQLite + sqlite-vec
- **云端**：Anthropic Claude API（项目已有 token 体系）
- **前端 (MVP)**：先 Web (Next.js / FastAPI + 简单 React)；移动端晚一步
- **Agent 编排框架**：自研轻量 orchestrator > LangChain（避免被框架绑死）

### v1（移动端）
- iOS：Swift + 本地 Whisper.cpp 转写 + 本地 Llama.cpp 推理
- Android：Kotlin，类似方案

## 8. 安全与隐私

1. **数据加密**：本地数据库加密（SQLCipher 或 iOS Keychain 派生密钥）
2. **云端最小化**：发到云端的只有"压缩后的事件摘要 + 信号"，不发原始语音
3. **可导出**：用户随时可导出全部数据为 markdown + JSON（防止锁定）
4. **可删除**：彻底删除时同步清云端缓存
5. **儿童内容红线**：所有云端调用都加 `safety_layer` 提示词，禁止生成不适合孩子的回应
