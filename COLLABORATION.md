# COLLABORATION.md — 协同接口

> 这是项目的"组织架构图"。读完它你就知道：每件事该找谁、产出在哪、怎么不打架。
> 任何新加入的 Agent / 人 / 会话，第一份要读的就是它。

---

## 1. 三方角色

### 1.1 你（瑶瑶爸爸）— 产品负责人 & 0号用户
- **唯一的人类决策者**：所有方向性、价值观、付费、对外发布相关的决定都由你拍板
- **数据来源**：所有关于瑶瑶的真实观察、反馈、感受
- **质量裁判**：只有你能判断"AI给的建议是否真的击中了"

### 1.2 Cowork（业务全岗 Claude）
**人设**：除了"写代码"以外的全部岗位都在这里。一人多岗，按需切角色。
**覆盖职能**（写明，避免后续误解）：
- **战略**：使命、愿景、长期方向、放弃条件
- **产品**：PRD 起草、需求分流、用户体验决策
- **运营**：内容运营、用户陪伴、内测节奏
- **数据分析**：指标定义、数据回顾、信号准确率评估
- **财务**：成本测算、$200/月 预算守门、定价模型
- **法务/合规**：未保条例、PIPL、数据出境、用户协议
- **营销 / GTM**：早期用户获取、定位、文案
- **反馈处理**：用户反馈分类、自动小改 vs 进 backlog 决策

**强项**：
- 和你来回聊清楚需求、目标、价值观
- 写 / 改战略文档（VISION/ROADMAP/PRD/差异化分析）
- 写 PRD，把需求翻译成 Code 能直接照做的需求卡
- 调用浏览器、文件、各类 MCP / Skill，做轻量编排和素材收集
**弱项**：
- 长链路、跨多文件、长时间编码会议不如 Code 稳定
- 不适合做几十文件的重构
- 写大段业务代码（应导出为 PRD 给 Code）

### 1.3 Claude Code(IDE/CLI)— 工程实现层
**人设**：拿到需求卡就闷头干活的资深工程师。
**强项**:
- 大型代码库导航与多文件一致性修改
- git 工作流、跑测试、IDE 集成
- 长编码会话稳定
**弱项**：
- 没看过你和 Cowork 的"对话历史"，需要靠文档接班
- 不应该独立做产品决策，遇到歧义要回到 Cowork

## 2. 两层 Agent 架构（重要，不要混淆）

### 2.1 第一层 — 开发期 Meta-Agent 层（我们怎么造这个产品）

```
你
 ├── Cowork (战略 / PRD / 反馈 / 文档)
 │     ↓ 产出 PRD、需求卡、ROADMAP 更新
 └── Claude Code (实现 / 测试 / 重构 / 发布)
       ↑ 反馈实现难点、提工程改进建议

           共享白板：
       /baby_grow_helper/   ← 文档 + 代码
       /memory/             ← 跨会话长期记忆
```

### 2.2 第二层 — 运行时 In-Product Agent 层（产品本身怎么跑）

```
用户(你/未来其他家长) 对话输入
 └── 记录Agent          ← 本地小模型，把口述/语音转成结构化条目
      └── 信号提取Agent ← 本地，每天/每周扫记录，标注兴趣/情绪/技能信号
           └── 洞察Agent ← 云端 Claude/GPT，按周/月生成天赋画像与解读
                └── 培养建议Agent ← 云端，按需触发，结合洞察+知识库给建议
                     └── [未来] 教练匹配 / 教育资源对接 Agent
```

**关键**：第一层是"我们怎么开发"，第二层是"产品本身怎么运转"。两者互不影响：你不会因为多请了一个 Code 实例帮你写代码就增加用户的运行成本。

## 4. 交接面 — 三个具体接口

### 接口 A：Cowork → Code 的"需求卡"

每个让 Code 去实现的任务，必须以 `prd/<short-name>.md` 形式存在，包含：

```markdown
# <Feature 名称>

## Why（为什么做，对应哪条使命/路线图）
## What（本次范围，明确不做什么）
## Acceptance（怎么算验收完成）
## Constraints（不能违反的约束：成本/隐私/依赖）
## Open Questions（Code 实现时若遇到歧义，回到 Cowork 问）
```

Code 模式启动时只需 `读 README.md → CLAUDE.md → ROADMAP.md → prd/<目标>.md` 就能开干。

### 接口 B：Code → Cowork 的回流

Code 在实现过程中会产生四类信息要回流：

1. **实际工作量与坑** → 更新 `ROADMAP.md` 的状态列
2. **架构层面的发现/反例** → 写一条 `decisions/NNNN-*.md`（ADR）
3. **超出 PRD 范围的歧义** → 在该 PRD 文件底部追加 `Open Questions`，等 Cowork 处理
4. **测试结果与性能数据** → 写到 `reports/` 里供后续洞察

### 接口 C：共享白板（两边都读都写）

| 路径 | 用途 | 读频率 |
|---|---|---|
| `README.md` | 项目入口，所有 Agent 第一份要读 | 每次新会话 |
| `CLAUDE.md` | Code 启动时自动加载的约束清单 | Code 每次会话 |
| `VISION.md` `DIFFERENTIATION.md` | 价值观底线，遇到模糊判断回查 | 决策歧义时 |
| `ARCHITECTURE.md` | 技术约束 | 写代码 / 改架构时 |
| `ROADMAP.md` | 当前在做什么 | 每次新会话 |
| `prd/` | 待实现需求 | Code 取活时 |
| `decisions/` | 历史决策与原因 | 想回溯"为什么这样做"时 |
| `memory/`（在 Cowork 端） | 长期偏好与个人风格 | Cowork 自动加载 |

## 5. 防打架的几条红线

1. **Code 不写战略文档**。如果 Code 想改 VISION/DIFFERENTIATION，必须先回 Cowork 走讨论。
2. **Cowork 不直接写大段业务代码**。轻量 demo / 脚本 / 一两文件以下可以，超过这个量级要导出 PRD 给 Code。
3. **谁都不修改 `memory/` 里别人写的条目**，要更新就追加、要废止就标记 `[deprecated]`。
4. **决策必须留痕**。任何"为什么这么做"的对话结论，必须沉淀到 `decisions/` 或对应文档里——不能只活在某一次对话里。
5. **大改动需 PR-style review**：Code 完成一个 PRD 后，要在 ROADMAP 里标 ✅ + 链接 PRD + 链接 commit；用户和 Cowork 抽查。

## 6. 升级路径（未来想引入更多 Agent 时怎么办）

如果未来要加新角色（例如 "市场Agent"、"教练匹配Agent"、"客服Agent"），按以下顺序加：

1. 在本文件第 1 节加一个角色卡（人设 / 强项 / 弱项）
2. 在第 3 节决策矩阵里加一行
3. 如果它和已有角色有交接，加一节 "接口 D / E …"

不要悄悄加 Agent。每加一个就更新一次本文件。
