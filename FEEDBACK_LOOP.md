# FEEDBACK_LOOP.md — 反馈到迭代的闭环

> 目标：让你的每一条反馈都不被浪费，但也不会让 AI 误改坏产品。
> 原则："小改自动、大改人工"——边界写死。

---

## 1. 反馈来源

1. APP 内反馈按钮（每条洞察/建议旁边的 👍/👎/✏️）
2. 全局意见入口（"想说点什么"）
3. 你和 Cowork 在对话里的临时吐槽（"上周这条建议太老套了"）—— Cowork 负责把这类对话也吸进流程

---

## 2. 反馈分类（产品 Agent 自动判断）

每条反馈被分类到以下四档之一：

| 分类 | 描述 | 例子 | 处理方式 |
|---|---|---|---|
| **A. 文案** | 措辞/语气/标点 | "这句太书面了" | 自动改 |
| **B. 阈值/参数** | 数值参数微调 | "信号置信度阈值太低，弱信号也报" | 自动改 |
| **C. 逻辑** | 算法/流程层面 | "周报应该按主题分组而不是时间" | 进 backlog 人工评 |
| **D. 需求** | 想要新东西 | "希望有夫妻共用账户" | 进 backlog 人工评 |

**模糊判定**：当置信度 < 0.7 时，默认走人工。

---

## 3. 自动小改流程（A、B 类）

```
反馈 ──→ 产品 Agent 分类
       ──→ 如果 A/B：
            1. 定位相关代码 / 文案常量文件
            2. 生成 diff 草稿
            3. 跑本地测试
            4. 提交一个 PR-style 改动到 changes/auto/
            5. 通知用户 review
            6. 用户一键合并 / 拒绝
       ──→ 如果 C/D：
            写一份 prd-draft 进 prd/inbox/
            等用户评审
```

**关键约束**（写死给产品 Agent 的红线）：
- 一次自动改动最多 **修改 3 个文件 / 新增 100 行代码**——超过就视为大改
- 改动不允许触及 `VISION.md`、`ARCHITECTURE.md` §1-2、数据库 schema
- 必须有 git commit 记录与回滚路径
- 如果改完后任何已有测试挂了 → 自动回滚 + 转人工

---

## 4. 大改人工流程（C、D 类）

```
反馈 ──→ 写成 prd/inbox/<feature>.md（草稿）
      ──→ 你和 Cowork 评审：保留 / 调整 / 拒绝
      ──→ 通过的搬到 prd/<feature>.md，进 ROADMAP 排期
      ──→ Code 模式实现
      ──→ 关闭 inbox 草稿
```

PRD 草稿模板（产品 Agent 自动生成）：
```markdown
# <Feature 名> (DRAFT, from feedback #<id>)

## 触发反馈
"<原文>"

## 问题理解
<产品 Agent 的总结>

## 提议方向（≥2 个）
1. ...
2. ...

## 风险 / 是否冲突 VISION
<产品 Agent 的核查>

## 建议优先级
P0/P1/P2/Defer

## 待用户确认
- [ ] 选哪个方向
- [ ] 是否进当前 Roadmap
```

## 5. 反馈数据结构

```sql
feedback(
  id, source,                  -- in_app / chat / system
  target_kind, target_id,      -- 哪条 insight/recommendation
  raw_text, rating,            -- 1-5 / 👍👎
  category,                    -- A/B/C/D
  category_confidence,
  status,                      -- received / classified / auto_fix_pending /
                              -- auto_fix_done / prd_draft / archived
  linked_change_id, linked_prd_id,
  created_at, resolved_at
)
```

每周生成一次"反馈处理总览"，让你看到：
- 反馈总数
- 自动处理 vs 人工占比
- 平均响应时间
- 哪类反馈最多 → 提示产品方向调整

---

## 6. 防误判保护

为了防止产品 Agent 自动改坏：

1. **自动改动有冷却期**：合并后 7 天内如果出现"同一文案被多次反馈是错的"，自动 revert
2. **每月一次回顾**：把上月所有自动改动列出来，你抽查
3. **重要文案锁**：核心文案（如周报开头、价值观相关措辞）打上 `# i18n_locked` 标记，自动改不能动
4. **影子模式**：高风险类别的自动改动不直接合并，先在影子环境跑 1 周观察用户行为

---

## 7. 长期目标

最终我们希望反馈循环达到这个状态：

> 你给一条反馈 → 24 小时内要么"已自动改好你看一下"要么"我把它写成 PRD 草稿了请评审"。
> 你不会再遇到"反馈了但石沉大海"的体验。

这本身也是产品差异化——市面上没有任何育儿 APP 把反馈循环做到这个程度。
